1. Inter-process Communication (IPC)

Background: Independent vs. Cooperating Processes

In a modern operating system, processes are not isolated entities. While an Independent Process does not share any data and remains unaffected by others, Cooperating Processes are designed to interact. This interaction is fundamental for building complex systems where tasks are broken into sub-tasks (modularity), computation is accelerated by running parts in parallel (speedup), and information like shared files or databases needs to be accessible by multiple users. To manage this interaction, the OS provides Inter-process Communication (IPC) mechanisms.

Primary Methods of IPC

The OS manages the physical distance between processes by providing two distinct architectural models. Each model addresses the core challenge of how to move data from the address space of one process to another safely and efficiently.

A. Shared Memory Model

In this model, the operating system allocates a specific segment of physical memory that is mapped into the address space of multiple participating processes. Once the segment is established, processes treat it as standard memory, reading and writing data directly without kernel intervention. This is widely considered the fastest IPC method because it avoids the overhead of repeated system calls and context switches.

Advantages:

  • Maximum communication speed (RAM speed).
  • Efficient for massive data transfers.
  • Kernel is only involved in setup.

Disadvantages:

  • Programmer must handle synchronization.
  • High risk of Race Conditions.
  • Difficult to implement across networks.

B. Message Passing Model

This model establishes a communication facility where processes exchange information via discrete packets or "messages." The OS kernel acts as an intermediary, managing a message queue (mailbox). Processes use system calls like send(message) and receive(message) to interact. This method inherently handles synchronization because the kernel ensures that messages are delivered in order and aren't corrupted.

Advantages:

  • Easier to program (sync is built-in).
  • Ideal for distributed/network systems.
  • Safer; no shared address space risk.

Disadvantages:

  • Slower due to kernel context switching.
  • Limited by system call overhead.
  • Less efficient for very large data.
Exam Version (Classification of Message Passing)
Message Passing can be further classified by:
  • Direct Communication: Processes must explicitly name each other (e.g., send(P, message)).
  • Indirect Communication: Messages are sent to and received from shared "mailboxes" or "ports."
  • Synchronization: Can be Blocking (Synchronous—sender waits until message is received) or Non-blocking (Asynchronous—sender sends and continues immediately).

2. Synchronization: The Race Condition & CS Problem

The Race Condition

A Race Condition is a critical vulnerability that occurs when multiple processes access and manipulate the same shared data concurrently. The final outcome of the program depends on the specific, unpredictable order in which the CPU interleaves low-level machine instructions. This happens because high-level code is not "Atomic"—it can be interrupted halfway through execution.

Machine-Level Analysis

Consider a shared variable count = 5. If two processes increment and decrement it simultaneously, the CPU breaks count++ into three steps: 1. Load count into register, 2. Add 1, 3. Store back. If a context switch occurs after Step 2, the second process reads the "old" value (5). When the first process eventually resumes, it saves its calculated value (6), completely overwriting the decrement done by the other process. This results in Data Inconsistency.

The Critical Section Problem

To solve the race condition, we identify the Critical Section (CS). This is the segment of code where a process modifies shared resources. The core challenge is to ensure that while one process is in its CS, no other process is allowed to enter theirs. A valid solution to this problem is evaluated based on three mandatory mathematical requirements.

The 3 Requirements for a Solution (8-10 Mark Question)

  • Mutual Exclusion: If process $P_i$ is executing in its critical section, then no other processes can be executing in their critical sections. This ensures exclusive access.
  • Progress: If no process is in its critical section and some processes wish to enter, the selection of the next process to enter cannot be postponed indefinitely. Only processes currently requesting entry can participate in the decision.
  • Bounded Waiting: There must be a limit (a bound) on the number of times other processes can "cut in line" after a process has made a request. This prevents Starvation.
Exam Version (Structural Components of CS Code)
A process managing a critical section follows a standard 4-part structure:
  1. Entry Section: The code that requests permission to enter the CS (e.g., acquiring a lock).
  2. Critical Section: The code where shared data is actually manipulated.
  3. Exit Section: The code that releases permission, allowing others to enter.
  4. Remainder Section: All non-shared, non-critical code.

3. Software and Hardware Solutions

Peterson's Solution (Software Logic)

Peterson's solution is a classic software-based algorithm for two processes ($P_0$ and $P_1$) to synchronize without special hardware. It uses two shared variables: a boolean array flag[2] (representing interest) and an integer turn (the tie-breaker). When $P_i$ wants to enter, it sets flag[i] = true and politely sets turn = j. It then waits in a busy-wait loop. It will only enter the CS if either the neighbor is not ready (flag[j] == false) or it is its own turn (turn == i). This solution is historically significant because it satisfies all three CS requirements through logic alone.

Hardware Synchronization (Atomic Instructions)

Software solutions like Peterson's are difficult to adapt to modern multi-core systems. Modern hardware provides Atomic Instructions, which are specialized CPU commands guaranteed to execute as a single, uninterruptible pulse. These instructions allow us to build efficient locking mechanisms.

  • 1. TestAndSet: This instruction reads the current value of a boolean variable and sets it to "true" in one atomic step. If it returns false, the process successfully "grabbed" the lock. If it returns true, the lock was already held, and the process must wait.
  • 2. CompareAndSwap (CAS): CAS is a more flexible instruction. It updates a memory location ONLY if the current value matches an "expected" value provided by the process. This prevents a process from accidentally updating data that changed behind its back during a context switch.

4. Semaphores and Process Coordination

The Semaphore Mechanism

A Semaphore is a high-level synchronization primitive that represents an integer variable accessed only through two atomic operations: wait() (P) and signal() (V). Unlike boolean locks, semaphores can manage complex access patterns and resource pools.

Types of Semaphores

  • Binary Semaphore: The integer value is restricted to 0 or 1. It behaves exactly like a Mutex Lock, providing mutual exclusion for a single resource.
  • Counting Semaphore: The value can range over an unrestricted domain. It is used to manage a pool of identical resources (e.g., a buffer with 10 slots). The count represents the number of available instances.

Working: Busy Waiting vs. Blocking

Primitive semaphores used Busy Waiting (Spinlocks), where a process would loop endlessly until the semaphore became positive. This consumes significant CPU cycles. Modern OS implementations use Blocking. If a process calls wait() and the resource is unavailable, the process is suspended (block()) and put in a wait queue. When another process calls signal(), the OS picks a process from the queue and wakes it up (wakeup()), moving it back to the ready queue.

5. Deadlock Characterization

System Model and Concept

A Deadlock is a state of total stagnation where every process in a set is waiting for an event (usually a resource release) that can only be triggered by another process in that same set. It creates a circular dependency where no progress is possible. In a standard system model, a process utilizes a resource in three steps: Request, Use, and Release. Deadlocks occur when multiple processes successfully "Request" some resources but cannot "Use" them because they are waiting on others.

The 4 Necessary Conditions (8-Mark Answer)

For a deadlock to occur, these four conditions must hold simultaneously. If you can break even one, deadlock is impossible.

  1. Mutual Exclusion: Resources are non-sharable; only one process can use an instance at a time.
  2. Hold and Wait: A process must be holding at least one resource while simultaneously waiting to acquire additional resources held by others.
  3. No Preemption: Resources cannot be forcibly taken away from a process; they must be released voluntarily by the owner.
  4. Circular Wait: A set $\{P_0, P_1, ..., P_n\}$ exists such that $P_0$ waits for $P_1$, $P_1$ waits for $P_2$, and $P_n$ waits for $P_0$.
Exam Version (RAG vs. Wait-for Graph)
  • Resource-Allocation Graph (RAG): A directed graph representing processes (circles) and resource types (squares). Arrows from process to resource indicate Request; arrows from resource to process indicate Assignment.
  • Wait-for Graph (WFG): A simplified version used for single-instance resources. Resource nodes are removed, and arrows are drawn directly between processes ($P_i \to P_j$ if $P_i$ waits for $P_j$).
  • Rule: A Cycle in a WFG is a definitive proof of deadlock.

Phase 2: Calculating Total Instances and Need

Step 1 Calculate Total Instances: Sum the "Allocation" column for each resource and add the "Available" value.

Example for Resource A: (0 + 1 + 1 + 0) + 1 (Avail) = 3 Total Instances.

Step 2 Fill the Need Matrix: Use the formula Need = Max - Allocation.

  • P0 Need: (0-0, 0-0, 1-1, 2-2) = (0, 0, 0, 0)
  • P1 Need: (1-1, 7-0, 5-0, 0-0) = (0, 7, 5, 0)
  • P2 Need: (2-1, 3-3, 5-5, 6-4) = (1, 0, 0, 2)
  • P3 Need: (0-0, 6-6, 5-3, 2-2) = (0, 0, 2, 0)

Phase 3: Finding the Safe Sequence

Step 3 Starting with Available (1, 5, 2, 0), check which process satisfies Need <= Available.

Check P0:

Need (0,0,0,0) ≤ Available (1,5,2,0). YES!

Action: P0 runs, finishes, and returns its Allocation (0,0,1,2).

New Available = (1,5,2,0) + (0,0,1,2) = (1, 5, 3, 2)

Check P1:

Need (0,7,5,0) ≤ Available (1,5,3,2). NO! (P1 must wait).

Check P2:

Need (1,0,0,2) ≤ Available (1,5,3,2). YES!

Action: P2 finishes, returns (1,3,5,4).

New Available = (1,5,3,2) + (1,3,5,4) = (2, 8, 8, 6)

Conclusion: A safe sequence exists, e.g., <P0, P2, P3, P1>. The system is in a SAFE STATE.

7. Management: Prevention, Detection, Recovery

Deadlock Prevention Strategies

Prevention aims to break at least one of the 4 conditions. While effective at stopping deadlocks, it often leads to low resource utilization.

  • Eliminating Hold and Wait: Require processes to request all resources at once. If it can't get all, it gets none.
  • Eliminating No Preemption: If a process holding resources is denied a new request, its current resources are taken away and given to others.
  • Eliminating Circular Wait: Assign a numerical order to every resource type. Processes must request resources only in strictly increasing order.
Deadlock Recovery

When detection algorithms (like Wait-for Graphs) identify a cycle, the OS must recover. This is achieved via two main approaches.

A. Process Termination

The OS can Abort All deadlocked processes, which is fast but wastes all work done. Alternatively, it can Abort One at a Time, checking for deadlock after each kill until the cycle is broken. This minimizes work loss but increases system overhead.

B. Resource Preemption

The OS forcibly takes a resource from a "Victim" process. This requires: 1. Victim Selection (choosing the process with minimum cost to kill), 2. Rollback (returning the victim to a safe state using checkpoints), and 3. Starvation Prevention (ensuring the same process isn't always chosen as the victim).
