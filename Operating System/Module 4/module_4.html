Part 1 & 2: Basic Concepts, Virtual Memory, and Exam Numericals

1. Basic Concepts: Swapping & Allocation

A. Swapping

Main memory (RAM) is limited. If a system wants to execute more processes than it has RAM for, it uses a technique called Swapping. The Operating System temporarily moves a process out of main memory and writes it to a secondary storage device (the hard disk or "Backing Store"). Later, when the CPU is ready to execute that process again, it swaps it back into RAM.

Pros: Allows a higher degree of multiprogramming and frees up memory for urgent processes.
Cons: Hard disks are drastically slower than RAM. Constant swapping can cause severe execution delays.

Swapping processes between Main Memory and Disk

B. Memory Allocation Strategies

When a process enters the system, the OS must allocate it a continuous block of memory. This is called Contiguous Memory Allocation. To find the right spot, the OS uses one of the following search algorithms:

  • First Fit: Allocates the first hole that is big enough. It stops searching immediately. Fast, but can leave awkward gaps.
  • Next Fit: Operates like First Fit, but starts searching from the location of the last allocation rather than the beginning.
  • Best Fit: Searches the entire list and allocates the smallest hole that is big enough. It produces the smallest leftover partition.
  • Worst Fit: Searches the entire list and allocates the largest available hole. Strangely, this is often better than Best Fit because it leaves behind a large leftover gap that can be used by another process later, whereas Best Fit leaves tiny, useless gaps.
Exam Numerical: First Fit vs. Best Fit Proof

The Problem Scenario (From Slides):

We have 3 available memory blocks: 150K, 150K, 350K.

Processes request memory in this order: 300K, 25K, 125K, 50K.

Why Best Fit FAILS:

  1. 300K process requests memory. Best Fit puts it in the 350K block. (Remaining: 50K).
  2. 25K process requests memory. The smallest block that fits is the new 50K block. (Remaining: 25K).
  3. 125K process requests memory. It fits into a 150K block. (Remaining: 25K).
  4. 50K process requests memory. FAILURE! The remaining free blocks are 25K, 25K, and 150K. The 50K process cannot fit into the 25K gaps, even though total free memory is 200K. This is External Fragmentation.

Why First Fit SUCCEEDS:

  1. 300K goes to 350K block. (Remaining: 50K).
  2. 25K goes to the first 150K block. (Remaining: 125K).
  3. 125K goes exactly into the 125K block left over from step 2. (Remaining: 0K).
  4. 50K goes exactly into the 50K block left over from step 1. (Remaining: 0K). ALL PROCESSES ALLOCATED!

Understanding Fragmentation

Internal Fragmentation: Occurs when memory is divided into fixed-size blocks. If you assign a 25KB process to a rigid 50KB block, the 25KB inside that block is completely wasted. It is "internal" to the block.

External Fragmentation: Occurs in dynamic allocation. As processes load and exit, they leave behind scattered gaps of free memory. Even if the total free memory is 100KB, a new 80KB process cannot be loaded because the free space is not contiguous (it's broken into tiny pieces). Solution: Compaction (shuffling memory to group free space) or Paging.

2. Paging, Page Tables, and the TLB

To completely eliminate External Fragmentation, Operating Systems use a non-contiguous memory allocation technique called Paging.

  • The Concept: The physical memory (RAM) is divided into fixed-sized blocks called Frames. The logical memory (the user process) is divided into blocks of the exact same size called Pages.
  • Because they are the exact same size, any Page can be loaded into any available Frame, no matter where it is located in RAM. No memory is wasted between blocks!

Address Translation Hardware

When the CPU executes code, it generates a Logical Address. The physical RAM does not understand logical addresses. The Memory Management Unit (MMU) uses a Page Table to translate the logical address into a Physical Address.

A Logical Address consists of two parts: 1. Page Number (p) (used as an index to the Page Table), and 2. Page Offset (d) (the exact location inside the page/frame). The MMU looks up the Page Number to find the matching Frame Number, and attaches the identical Offset to generate the physical address.

Exam Numerical: Paging Address Calculation (Decimal)

Problem: Given a logical address of 21205 and a Page Size of 1024 bytes (1 KB), find the Page Number and Offset.

Formulas:

  • Page Number = Logical Address / Page Size
  • Offset = Logical Address % Page Size (Remainder)

Step-by-Step Solution:

1. Page Number = 21205 / 1024 = 20 (Take the whole number)

2. Offset = 21205 % 1024 = 725

Answer: Page Number is 20, Offset is 725.

Translation Lookaside Buffer (TLB)

Every time the CPU wants to access memory, it has to check the Page Table. Since the Page Table is stored in RAM, this requires two trips to memory (one for the table, one for the actual data), cutting performance in half! To fix this, we use a TLB.

The TLB is a small, ultra-fast hardware cache placed inside the MMU. It stores the most recently used Page-to-Frame mappings. When a logical address is generated, the CPU checks the TLB first.

  • TLB Hit: The mapping is found in the TLB. Fast access!
  • TLB Miss: The mapping is not found. The CPU must access the Page Table in RAM, and then copies that mapping into the TLB for future use.
Exam Numerical: Effective Memory Access Time (EMAT)

Problem: Assume a TLB hit ratio of 80% (0.80). It takes 15ns to search the TLB, and 150ns to access main memory. Calculate the EMAT.

Formula:

EMAT = Hit Ratio * (TLB_time + Mem_time) + Miss Ratio * (TLB_time + Mem_time + Mem_time)

*Note on Miss: You search TLB (15ns), fail, search Page Table in RAM (150ns), then access the actual data in RAM (150ns).

Step-by-Step Solution:

1. Hit Case = 0.80 * (15 + 150) = 0.80 * 165 = 132 ns

2. Miss Case = 0.20 * (15 + 150 + 150) = 0.20 * 315 = 63 ns

3. EMAT = 132 + 63 = 195 ns

3. Segmentation

While Paging divides memory into fixed, meaningless mathematical blocks, Segmentation divides memory logically. A programmer doesn't think of memory as "Page 1 and Page 2". They think of it as "The Main Program", "The Stack", "The Math Library", and "The Variables".

Segmentation supports this Logical View. Memory is divided into variable-sized segments based on the actual components of the program. Because segments are different sizes, it suffers from External Fragmentation.

The Segment Table

Instead of a Page Table, the MMU uses a Segment Table. Because segments vary in size, the table must track two things for every segment:

  • Base: The exact starting physical address of the segment in RAM.
  • Limit: The length of the segment. If the CPU tries to access an offset that is larger than the Limit, the OS throws a "Segmentation Fault" error to prevent illegal memory access.

4. Virtual Memory, Demand Paging & Copy-on-Write

Basic Concepts of Virtual Memory

Virtual Memory is a technique that creates the illusion of a massive, continuous memory space, even if your physical RAM is small. It does this by treating a portion of your hard disk as an extension of RAM. Instead of loading an entire 4GB video game into RAM at once, the OS only loads the specific pages the game is actively using right now. The rest stay on the disk.

Demand Paging

This is the mechanism that makes Virtual Memory work. The OS uses a "Lazy Swapper" (or Pager). It never swaps a page into memory unless that page is actually demanded by the CPU. When a process starts, its Page Table is created, but all entries are set to an Invalid Bit (meaning the page is on the disk, not in RAM). If the CPU asks for a page and sees the Invalid Bit, a Page Fault occurs. The OS pauses the process, fetches the page from the disk, puts it in a free RAM frame, updates the table to "Valid", and resumes the process.

Copy-on-Write (CoW)

Copy-on-Write is a brilliant optimization used during process creation (like the `fork()` system call). When Process A creates a child Process B, normally the OS would duplicate the entire memory of A into B. This is terribly slow and wastes RAM. With CoW, both Process A and Process B simply point to the exact same physical pages in RAM! They share the memory.

The pages are marked as "Copy-on-Write". They only share the memory as long as they are just reading it. The moment Process B attempts to modify (write to) a variable, the OS intercepts it, creates a duplicate copy of just that specific page for Process B, and allows the write. This saves massive amounts of time and memory.

5. Page Replacement Algorithms & Thrashing

What happens if a Page Fault occurs, but the physical RAM is completely full? The OS must perform Page Replacement. It evaluates the frames in memory, chooses a "victim" frame, swaps it out to the disk, and loads the newly requested page in its place. The goal of every algorithm is to choose a victim that minimizes future page faults.

Mechanics: Page Fault Counter & Victim Selection

Every time the CPU needs to read or write data, it requests a specific Page. The OS follows a strict sequence to handle this request and updates the Page Fault Counter accordingly:

  1. The Request & Check: The OS checks if the requested page is already loaded into one of the available physical RAM frames.
  2. Case A: Page Hit (No Action)
    • If the page is found in RAM, it's a Hit.
    • The Page Fault Counter DOES NOT INCREASE.
    • No victim is selected, and execution continues instantly.
  3. Case B: Page Fault / Miss (Counter Increases)
    • If the page is missing from RAM, a Page Fault (Miss) occurs.
    • The OS immediately pauses the process, and the Page Fault Counter increases by +1.
  4. Resolving the Fault (Frame Allocation & Victim Selection): The OS must now fetch the missing page from the hard disk into RAM.
    • Condition 1 (Empty Frames Available): If there is an empty frame (usually at the start of execution), the page is simply placed in the empty frame. No victim is needed.
    • Condition 2 (RAM is Full): If all frames are occupied, the OS triggers the Victim Selection Logic. Based on the specific algorithm (FIFO, LRU, etc.), the OS chooses an existing page to be "evicted" (overwritten). The newly requested page replaces this victim.

Exam Rule: Total Page Faults = (Initial Empty Frame Fills) + (Number of Victim Replacements)

⚠️ Belady's Anomaly

Normally, common sense dictates that adding more physical memory (Frames) should decrease the number of Page Faults. Belady's Anomaly is a counter-intuitive phenomenon where increasing the number of page frames actually increases the number of page faults!

Example Proof:

Given the reference string: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5

  • Using FIFO with 3 Frames yields exactly 9 Page Faults.
  • Using FIFO with 4 Frames yields exactly 10 Page Faults!

How to solve Belady's Anomaly?

Belady's Anomaly strictly affects the FIFO algorithm. To solve it, we use Stack Algorithms (like LRU or Optimal). Stack algorithms mathematically guarantee that the set of pages in memory for N frames is always a strict subset of the pages in memory for N+1 frames. Thus, increasing frames can never increase faults.

The simplest algorithm. The OS uses a queue to track pages. When replacement is needed, it simply evicts the oldest page (the one brought into memory first). It performs poorly because an old page might still be heavily used.

Thrashing

If a system does not have enough physical frames to support the active processes, a process will constantly suffer page faults. The OS will spend all its time rapidly swapping pages in and out of the slow disk, instead of actually executing instructions. This severe performance collapse is called Thrashing.

Thrashing causes a massive drop in CPU utilization. The OS sees low CPU usage, thinks the system is idle, and tries to load even more processes into memory, making the problem exponentially worse!

As multiprogramming increases, CPU utilization drops sharply due to Thrashing.

How to solve Thrashing? The OS uses the Working Set Model. It observes the "Locality" of a process (the group of pages it is actively using right now) and ensures the process is allocated enough frames to hold its entire working set. If total memory demand exceeds total available frames, the OS suspends an entire process to free up memory.
