Operating System: Module 2 (Process Management)

1. Basic Concepts of Process & State Transition

The Process: A process is a program in execution. It needs resources (CPU time, memory, files, I/O devices) to accomplish its task. The status of its current activity is represented by the program counter and processor registers.

Memory Layout of a Process:

  • Text section: The executable code.
  • Data section: Global and static variables (initialized and uninitialized).
  • Heap section: Memory dynamically allocated during program run time.
  • Stack section: Temporary data storage when invoking functions (function parameters, return addresses, local variables).
Figures: Layout of a process in memory / Memory layout of a C program showing Stack and Heap growing towards each other.

Process State Transition Model:

A process goes through different states during execution. The State Transition Diagram visually represents these transitions:

  • New: The process is being created (Admitted into the Ready queue).
  • Ready: The process is waiting to be assigned to a processor (Waiting for CPU scheduling).
  • Running: Instructions are being executed (CPU is allocated).
  • Waiting (Blocked): The process is waiting for some event to occur (e.g., I/O completion or reception of a signal). Once I/O is complete, it moves back to Ready.
  • Terminated: The process has finished execution or is forcefully terminated.
Figures: Diagram of process state (New, Ready, Running, Waiting, Terminated) / 7-Process States Diagram (including Suspended Ready & Suspended Blocked).

2. Operations on Processes & PCB

Process Creation:

  • A process (parent) may create several new processes (children), forming a tree of processes.
  • Identified by a unique process identifier (pid). (Example: Linux systemd process has pid = 1, serving as the root parent process for all user processes).
  • Execution Possibilities: 1. Parent executes concurrently with children. 2. Parent waits until children terminate.
  • Address Space Possibilities: 1. Child is a duplicate of the parent (same program/data). 2. Child has a new program loaded into it.
  • UNIX Example: The fork() system call creates a child process (child pid=0, parent gets child's actual pid). The child inherits privileges/open files. The child overlays its address space using exec(). The parent waits using wait() and resumes when the child invokes exit().
Figures: A tree of processes on a typical Linux system / Process creation using the fork() system call.

Process Termination:

  • A process terminates when it finishes its final statement and asks the OS to delete it using the exit() system call. Returns status to waiting parent via wait().
  • All resources (physical/virtual memory, open files, I/O buffers) are deallocated.
  • A parent may terminate a child if:
    • Child exceeded resource usage.
    • Task assigned to child is no longer required.
    • Parent is exiting, and OS does not allow a child to continue (Cascading termination).

Process Control Block (PCB):

Each process is represented in the OS by a PCB (Task Control Block). It serves as the repository for all data needed to start/restart a process:

  • Process state: New, ready, running, waiting, etc.
  • Program counter: Address of the next instruction to be executed.
  • CPU registers: Accumulators, index registers, stack pointers. Saved during interrupts to continue correctly later.
  • CPU-scheduling info: Priority, pointers to scheduling queues.
  • Memory-management info: Base/limit registers, page tables.
  • Accounting info: CPU time used, time limits, process numbers.
  • I/O status info: List of I/O devices allocated, list of open files.
Figure: Process control block (PCB).

3. Context Switching

Definition: The process where the CPU stops running one process, saves its current state in its PCB, and loads the saved state of another process so that multiple processes can share the CPU effectively.

  • Need in Multitasking: CPU keeps switching between processes, giving the illusion of parallel execution. Without it, one process could monopolize the CPU indefinitely.
  • Role in Scheduling: The scheduler decides the next process; the dispatcher performs the actual switch.
  • Triggers for Context Switch: High-priority process arrives, Interrupt occurs, User/Kernel mode switch, or Preemptive CPU scheduling ends a time slice.
  • Overhead: Time spent switching is called context switch overhead. The CPU does no useful work during this time. If switching happens too often, efficiency drops.

Step-by-Step Execution Phase (P0 to P1):

  1. P0 is executing. Interrupt or system call occurs.
  2. Save State (P0 → PCB0): OS pauses P0 and saves its register values/PC into PCB0.
  3. Load State (PCB1 → P1): OS retrieves the saved state of P1 from PCB1.
  4. P1 starts executing while P0 remains idle.

4. Introduction to Threads & Thread Models

Thread: A single sequence stream within a process. Often called a "lightweight process." It is a basic unit of CPU utilization comprising a Thread ID, Program Counter (PC), Register set, and Stack.

Multiple threads within the same process share the code section, data section, and OS resources (like open files).

Process vs. Thread (6 Key Differences):

  • Weight: Process is heavy-weight/resource-intensive. Thread is light-weight, using fewer resources.
  • Switching: Process switching needs interaction with the OS. Thread switching does not need OS interaction.
  • Resources: Each process has its own memory/file resources. Threads share the same set of open files/child processes.
  • Blocking: If one process blocks, no other process can execute. If one thread blocks, a second thread in the same task can still run.
  • Resource Usage: Multiple processes use more resources; multithreaded processes use fewer resources.
  • Independence: Processes operate independently. One thread can read, write, or change another thread's data.
Figure: Single-threaded process vs multithreaded process layout.

Types of Threads:

  • User-Level Thread (ULT): Implemented in user-level libraries. Kernel doesn't know about them.
    Pros: Fast context switching, portable, no kernel intervention needed.
    Cons: If one thread causes a page fault/blocking system call, the entire process blocks.
  • Kernel-Level Thread (KLT): Managed directly by the OS. Kernel has a master thread table.
    Pros: True parallel execution on multiprocessors; if one thread blocks, others can run.
    Cons: Slower and inefficient context switching due to OS intervention overhead.

Multithreading Models:

  • Many-to-One: Maps many ULTs to one KLT. Efficient, but entire process blocks if one thread blocks. Cannot run in parallel on multicore systems.
  • One-to-One: Maps each ULT to a KLT. Provides maximum concurrency. Drawback: creating a user thread requires creating a kernel thread, burdening performance (used by Windows, Linux).
  • Many-to-Many: Multiplexes many ULTs to a smaller/equal number of KLTs. Best of both worlds. (A variation is the Two-level model which also allows a ULT to be strictly bound to a KLT).
Figures: User Level Thread vs Kernel Level Thread / Many-to-One Model / One-to-One Model / Many-to-Many Model / Two-level Model.

5. Basic Concepts of Scheduling & Types of Schedulers

Basic Concepts: Process scheduling manages removing a running process from the CPU and selecting another to maximize CPU utilization. A process moves between queues (Job queue, Ready queue, Waiting queue) throughout its lifetime.

Types of Schedulers:

  • 1. Long-Term Scheduler (Job Scheduler):
    Loads processes from disk (Job Queue) into main memory (Ready Queue).
    Function: Controls the degree of multiprogramming. Selects a balanced mix of I/O-bound and CPU-bound processes.
    Speed: Slowest. Operates infrequently. (Almost minimal/nil in modern time-sharing systems).
  • 2. Short-Term Scheduler (CPU Scheduler):
    Selects a process from the ready queue and assigns the CPU to it.
    Function: Maximizes CPU utilization and ensures no starvation. Uses scheduling algorithms.
    Speed: Fastest. Operates very frequently (every few milliseconds).
  • Dispatcher: The component called by the Short-Term Scheduler that performs the actual context switch, switches system to user mode, and jumps to the proper program location.
  • 3. Medium-Term Scheduler:
    Manages swapping. Temporarily removes processes from memory to disk (Swap out) when they are blocked, and restores them later (Swap in).
    Function: Reduces the degree of multiprogramming to free up memory.
    Speed: Faster than Long-term, slower than Short-term.
Figures: Process scheduler queues / Short-Term Scheduler flowchart / Medium-Term Scheduler swapping flowchart.

6. Scheduling Algorithms & Criteria

A. Scheduling Definitions & Criteria

  • Non-Preemptive: Once a process gets the CPU, it holds it until it terminates or switches to waiting. It cannot be interrupted.
  • Preemptive: OS can interrupt a running process and force it into the ready state to allocate the CPU to another process.

Formulas & Criteria:

  • CPU Utilization: Keep CPU as busy as possible (40% to 90%).
  • Throughput: Number of processes completed per time unit.
  • Turnaround Time (TAT) = Completion Time (CT) - Arrival Time (AT)
  • Waiting Time (WT) = Turnaround Time (TAT) - Burst Time (BT)
  • Response Time = Time from submission until the FIRST response is produced.

B. Non-Preemptive Scheduling Algorithms

1. First-Come, First-Served (FCFS)

Managed with a FIFO queue. The process that requests the CPU first gets it first.

Disadvantage: High average waiting time. Suffers from the Convoy Effect: Short processes are forced to wait behind long processes, leading to inefficiency.

Example Calculation: P1(24), P2(3), P3(3). Order P1, P2, P3 → Avg WT = 17ms. Order P2, P3, P1 → Avg WT = 3ms.

Gantt Charts: FCFS scheduling examples

2. Non-Preemptive Shortest Job First (SJF)

CPU is assigned to the process that has the smallest next CPU burst. Once assigned, it runs to completion without interruption.

Advantage: Gives the minimum possible average waiting time for a given set of processes.

Gantt Charts: Non-Preemptive SJF

3. Non-Preemptive Priority Scheduling

CPU is allocated to the process with the highest priority (e.g., 0 = highest). Once it starts, it cannot be interrupted even if a higher priority process arrives.

Example from PPT: P1 (Burst 4, Priority 3), P2 (5, 2), P3 (8, 2), P4 (7, 1), P5 (3, 3).
Execution Order: P4 → P2 → P3 → P1 → P5.

Gantt Chart: Priority scheduling execution order

C. Preemptive Scheduling Algorithms

1. Round-Robin (RR) Scheduling

Designed for time-sharing systems. A small unit of time called a time quantum (10-100ms) is defined. The ready queue is a circular FIFO queue. Each process gets the CPU for up to 1 quantum. If the burst exceeds it, it is preempted and put at the tail of the ready queue.

Example: P1(24), P2(3), P3(3). Quantum = 4. P1 gets 4ms, preempted. P2 gets 3ms (quits). P3 gets 3ms (quits). P1 gets remaining. Avg WT = 5.66ms.

Gantt Chart: Round-Robin scheduling / Figure: Time quantum vs Context switches

2. Preemptive SJF (Shortest-Remaining-Time-First / SRTF)

If a new process arrives with a CPU burst shorter than the remaining time of the currently running process, the running process is preempted.

Example: P1(arr 0, bt 8), P2(arr 1, bt 4), P3(arr 2, bt 9), P4(arr 3, bt 5). Avg WT = 6.5ms.

Gantt Charts: Preemptive SJF (SRTF)

3. Preemptive Priority Scheduling

If a new process arrives with a higher priority than the currently running process, the CPU is preempted and given to the new arrival.

Major Problem: Starvation (Indefinite blocking). Low priority processes may wait indefinitely.

Solution: Aging. Gradually increasing the priority of processes that wait in the system for a long time. (Fact: It would take a little over 2 minutes for a priority-127 process to age to a priority-0 process.)

D. Multilevel Queue Scheduling

With both priority and round-robin scheduling, all processes may be placed in a single queue, requiring an O(n) search to determine the highest-priority process. In practice, it is easier to have separate queues for each distinct priority.

Priority scheduling simply schedules the process in the highest-priority queue. If there are multiple processes in the highest-priority queue, they are executed in Round-Robin order. In its most generalized form, a priority is assigned statically to each process, and it remains in the same queue for the duration of its runtime.
