Database Management System: Module 1

"A database is a set of related data. A Database Management System (DBMS) is a software system that allows users to access the data contained in the database, acting as an interface for insert, delete, update, and retrieval operations."

1. Introduction & Application Areas

Commercially available DBMS: Oracle, Sybase, Ingress, Microsoft Access, FoxPro, Dbase.

Key Application Areas:

  • Banking: Customer information, accounts, loans, and all banking transactions.
  • Airlines: Reservations, schedules, and flight information.
  • Universities: Student information, registrations, and grades.
  • Enterprise Information: Sales (customers, products, purchases), Accounting (payments, receipts, balances), Human Resources (employee records, salaries, payroll taxes).
  • Manufacturing: Supply chain management, production tracking, inventories, and orders.
  • Telecommunication: Call records, monthly bills, and prepaid balance maintenance.
  • Online Retailers: Online order tracking, recommendation lists, and product evaluations.

2. Drawbacks of File Processing System (Why use DBMS?)

Before DBMS, data was stored in traditional file systems, which had several major limitations:

  • 1. Data Redundancy and Inconsistency: Same information duplicated in multiple files (e.g., a student's address stored separately in Music and Math department files). Leads to higher storage costs and data inconsistency (updating one copy but not the other).
  • 2. Difficulty in Accessing Data: Requires writing a new application program for every specific data retrieval request. Cannot anticipate all future requests easily.
  • 3. Data Isolation: Data is scattered in various files and formats, making it difficult to write new applications to retrieve appropriate related data.
  • 4. Integrity Problems: Data must satisfy consistency constraints (e.g., account balance > 0). It is difficult to add or change constraints in application code, especially across different files.
  • 5. Atomicity Problems: A process must happen in its entirety or not at all. If a system fails during a $500 fund transfer, the money might be debited from Account A but not credited to Account B. File systems cannot guarantee atomicity.
  • 6. Concurrent-Access Anomalies: Multiple users updating data simultaneously can result in inconsistent data (e.g., two clerks debiting the same account balance simultaneously, overwriting each other's changes).
  • 7. Security Problems: Hard to enforce security constraints where users only see specific parts of data (e.g., payroll staff seeing financial data, but not academic records).

3. Data Models & Data Abstraction

A. Data Models

  • Relational Model: Uses a collection of tables (relations) to represent data and relationships. Each table contains fixed-format records (rows) and attributes (columns).
  • Entity-Relationship (E-R) Model: Uses a collection of basic objects (entities) and the relationships among them. Very widely used in database design.
  • Object-Based Data Model: Integrates object-oriented programming concepts (like Java/C++) into databases. Procedures can be stored and executed by the database system.

B. Levels of Data Abstraction

Developers hide complexity from users through several levels of abstraction:

  • 1. Physical Level (Lowest Level): Describes how data is actually stored in disk drives. Managed by the Database Administrator (DBA) who decides physical data fragmentation, centralization, and file structures.
  • 2. Logical / Conceptual Level: Describes what data is stored and its structures (blueprint). It defines the tables, columns (e.g., student_name, roll_no), data types, and relationships between tables.
  • 3. View Level (Highest Level): Describes how data is shown to the end user. Multiple views exist for security and simplicity (e.g., a student views only their marks, while faculty views all students' marks and salaries).

4. ACID Properties

Transactions are fundamental operations that modify/retrieve data. To maintain integrity, correctness, and reliability even during failures, transactions must follow ACID properties:

  • A = Atomicity ("All or nothing"): A transaction is all-or-nothing; either all its operations succeed (Commit), or none are applied (Abort/Rollback). If any part fails, the entire transaction is rolled back to keep the database consistent.
  • C = Consistency ("Valid State"): The database must remain in a valid state before and after a transaction. It must follow all defined rules, constraints, and relationships. If a rule is violated, the transaction rolls back. (e.g., Total bank balance before transfer = Total balance after transfer).
  • I = Isolation: Transactions run independently without affecting each other. Changes made by one transaction are not visible to others until committed. Ensures concurrent transactions yield the same result as sequential ones.
  • D = Durability: Once a transaction is committed, its changes are permanently saved in non-volatile memory. Even if the system crashes immediately after a commit, the data will survive.

5. DBMS Architecture

The DBMS Architecture is divided into interfaces for specific users (Naive users, Application programmers, Sophisticated users, DBAs) and two main internal components:

1. Query Processor

Simplifies and facilitates access to data. It parses and optimizes queries to produce an efficient execution blueprint.

  • DDL Interpreter: Translates Data Definition Language (DDL) statements and updates data dictionaries.
  • DML Compiler: Translates Data Manipulation Language (DML) queries into an evaluation plan of low-level instructions.
  • Query Evaluation Engine: Executes the low-level instructions generated by the DML compiler against the database.

2. Storage Manager

Provides the interface between low-level data stored in the database and the queries submitted. Responsible for storing, retrieving, and updating data.

  • Authorization and Integrity Manager: Tests for integrity constraints and checks user authority/access rights.
  • Transaction Manager: Ensures DB remains in a consistent state despite failures and handles concurrent transactions without conflict.
  • File Manager: Manages space allocation on disk storage and data structures representing information.
  • Buffer Manager: Crucial component that fetches data from disk to main memory (cache). Enables DBMS to handle databases much larger than main memory.

Storage Manager Data Structures:

  • Data files: Store the database itself.
  • Data dictionary: Stores metadata (schema of the database).
  • Indices: Provide fast access pointers to data items holding particular values.

6. The Entity-Relationship (ER) Model

Entity Types & Sets

  • Entity: A real-world object that is distinguishable (e.g., a specific Student or Car).
  • Entity Set: A collection of similar entities (e.g., All Students).
  • Strong Entity Set: Has its own primary key and can exist independently. Represented by a standard rectangle.
  • Weak Entity Set: Does not have a primary key of its own. Its existence depends on a strong entity (identifying entity). Represented by a double rectangle.
Feature Strong Entity Set Weak Entity Set
Primary Key Possesses its own primary key. Does not have a primary key (uses a partial key or discriminator instead).
Existence Independent of any other entity in the schema. Existence is strictly dependent on a strong entity (identifying entity).
Representation Represented by a single rectangle in ER diagrams. Represented by a double rectangle in ER diagrams.
Relationship Type Uses a standard relationship (represented by a single diamond). Uses an identifying relationship (represented by a double diamond).
Participation Participation in its relationship can be partial or total. MUST have total participation in the identifying relationship.

Types of Attributes

  • Simple Attribute: Cannot be further divided (e.g., Age).
  • Composite Attribute: Can be divided into smaller sub-parts (e.g., Name can be divided into First, Middle, Last).
  • Single-valued Attribute: Holds a single value for an entity (e.g., Date of Birth).
  • Multi-valued Attribute: Can hold multiple values (e.g., Phone Numbers). Represented by a double oval.
  • Derived Attribute: Value is derived from other attributes (e.g., Age derived from Date of Birth). Represented by a dashed oval.

Keys

  • Super Key: A set of one or more attributes that can uniquely identify a record.
  • Candidate Key: A minimal super key (no unnecessary attributes).
  • Primary Key: The specific candidate key chosen by the database designer to uniquely identify entities. Underlined in ER diagrams.
  • Foreign Key: An attribute that creates a relationship between two tables, referring to the primary key of another table.

Relationship Constraints

  • Mapping Cardinality: Defines the maximum number of relationship instances an entity can participate in.
    • One-to-One (1:1)
    • One-to-Many (1:N)
    • Many-to-One (N:1)
    • Many-to-Many (M:N)
  • Participation Constraint:
    • Total Participation: Every entity in the set must participate in the relationship (Double line).
    • Partial Participation: Only some entities participate in the relationship (Single line).

7. Extended Entity-Relationship (EER) Model

The EER model incorporates all ER model concepts but adds complex advanced features for better abstraction:

  • 1. Generalization: A bottom-up process. Multiple lower-level entities with common attributes are combined to form a higher-level generalized entity.
    Example: "Car" and "Truck" entities can be generalized into a "Vehicle" entity.
  • 2. Specialization: A top-down process. A higher-level entity is broken down into lower-level sub-entities based on distinguishing features. It is the exact opposite of generalization.
    Example: An "Employee" entity can be specialized into "Developer", "Tester", or "Manager".
  • 3. Aggregation: A relationship itself is treated as a higher-level entity so that it can participate in relationships with other entities. It overcomes the limitation of the ER model where relationships cannot have relationships with other entities.
    Example: If an employee works on a project (relationship "Works-On"), and requires machinery for that specific work, the "Works-On" relationship is aggregated into an entity to connect it to "Machinery".
