Database Management System: Module 2

1. Introduction to Relational Model & Schema

The Relational Model represents data as a collection of two-dimensional tables called Relations. It is the most widely used data model due to its simplicity and mathematical foundation.

Core Terminologies:

  • Relation: A mathematical term for a Table (e.g., Student table).
  • Tuple: A single row (or record) in a relation representing a specific real-world entity instance.
  • Attribute: A column (or field) in a relation representing a property of the entity.
  • Domain: The set of allowable, valid values for a specific attribute (e.g., Domain of 'Age' is numbers between 1 and 120).
  • Degree: The total number of attributes (columns) in a relation.
  • Cardinality: The total number of tuples (rows) in a relation.

Relational Schema vs. Relational Instance:

  • Relational Schema: The logical design/blueprint of the relation. It defines the table name, its attributes, and their domains.
    Format: Relation_Name(Attribute1, Attribute2, ..., AttributeN)
    Example: Student(RollNo, Name, Email)
  • Relational Instance: The actual data/set of tuples stored in the relation at a specific moment in time. It changes dynamically as data is inserted or deleted.

2. Concept of Keys

Why do we need keys? In a database table, many records look similar. Keys help us uniquely identify each record in a relation, just like identification cards identify people.

  • 1. Super Key: Any set of one or more attributes that can uniquely identify a record in a table. A super key may contain extra/unnecessary attributes, but uniqueness is guaranteed.
    Student Table Example: Attributes are {StudentID, RollNo, Email, Name}.
    Possible Super Keys: {StudentID}, {RollNo}, {Email}, {StudentID, Name}, {RollNo, Email}. Even if we add the unnecessary "Name" attribute to "StudentID", the combination remains unique.
    Bank Customer Example: {AccountNumber}, {AccountNumber, CustomerName}, {AccountNumber, BranchName} are all Super Keys.
  • 2. Candidate Key: A minimal super key. "Minimal" means if you remove even a single attribute from this key, uniqueness is lost. It contains absolutely no extra attributes.
    Example: {StudentID}, {RollNo}, and {Email} are individual Candidate Keys. However, {StudentID, Name} is a Super Key but not a Candidate Key because "Name" is extra.
  • 3. Primary Key: The specific Candidate Key chosen by the database designer to uniquely identify tuples within the relation. There can be only ONE Primary Key per table. It cannot contain NULL values.
    Example: Out of Candidate Keys {StudentID} and {Email}, the designer chooses {StudentID} as the Primary Key.
  • 4. Foreign Key (Referential Integrity): An attribute (or set of attributes) in one table that refers to the Primary Key of another table. It is used to link two tables together and enforce referential integrity constraints.
Table Example: Student(StudentID, RollNo, Email, Name) | 101, 21, a@gmail.com, Ravi | 102, 22, b@gmail.com, Meena

3. Mapping ER and EER Model to Relational Model

To convert an ER/EER diagram into Relational Tables (Schemas), specific mapping rules are strictly followed:

A. Mapping ER Entities & Attributes

  • Strong Entity: Create a new table. Each simple attribute becomes a column. The chosen key attribute becomes the Primary Key.
  • Weak Entity: Create a new table. Add all its simple attributes. Add the Primary Key of the identifying strong entity as a Foreign Key. The new Primary Key is the combination of the partial key (discriminator) and the foreign key.
  • Composite Attribute: Do not create a column for the composite attribute itself. Instead, create separate columns for each of its simple sub-components (e.g., use First_Name and Last_Name columns instead of Name).
  • Multivalued Attribute: Create a entirely new table. This table contains the multivalued attribute and the Primary Key of the parent entity as a Foreign Key. The Primary Key of this new table is the combination of both.

B. Mapping ER Relationships (Cardinalities)

  • 1:N (One-to-Many): Do not create a new table. Take the Primary Key of the "1" side and put it as a Foreign Key in the table of the "N" (many) side.
  • 1:1 (One-to-One): Put the Primary Key of one entity into the other entity's table as a Foreign Key. (Preferably put it in the table that has Total Participation).
  • M:N (Many-to-Many): Must create a new table for the relationship. This new table's Primary Key is a combination of the Primary Keys of both participating entities (which act as Foreign Keys).

C. Mapping EER (Generalization & Specialization)

  • Cross-Table Approach: Create one table for the higher-level (super-class) entity containing all common attributes and a Primary Key. Create separate tables for each lower-level (sub-class) entity. Put the specific attributes in these tables, and add the Primary Key of the super-class as a Foreign Key (which also acts as their Primary Key).

4. Relational Algebra - Operators

Relational Algebra is a procedural query language. It takes one or two relations as input and produces a new relation as output.

A. Unary Operators (Work on 1 Table)

  • Selection (σ): Selects specific Rows (tuples) from a relation that satisfy a given condition.
    Syntax: σcondition(Relation)
    Example: σsalary > 50000(Employee) (Selects all rows where salary is greater than 50k).
  • Projection (π): Selects specific Columns (attributes) from a relation and discards the rest. It automatically eliminates duplicate rows.
    Syntax: πattribute_list(Relation)
    Example: πName, Salary(Employee) (Shows only the Name and Salary columns).
  • Rename (ρ): Used to rename the output relation or its attributes.
    Syntax: ρNew_Name(Relation)

B. Set-Theoretic Binary Operators (Work on 2 Tables)

Note: For Union, Intersection, and Difference, the two tables must be "Union Compatible" (same number of columns and matching data types).

  • Union (∪): Combines all tuples from two relations into a single relation. Duplicate tuples are eliminated.
    Syntax: A ∪ B (Returns tuples in A, or in B, or in both).
  • Intersection (∩): Returns a relation containing only the tuples that are present in both relations.
    Syntax: A ∩ B
  • Set Difference (−): Returns tuples that are present in the first relation but not in the second relation.
    Syntax: A − B (Tuples in A but not in B).
  • Cartesian Product (×): Combines information from any two relations. It pairs every single row of the first table with every single row of the second table. (If A has 5 rows and B has 4 rows, A × B will have 20 rows).
    Syntax: A × B

C. Derived Operators (Joins)

  • Join (⨝): A Cartesian Product followed by a Selection process. It combines related tuples from two tables into a single tuple based on a common attribute.
  • Types of Joins:
    • Theta Join (⨝θ): Joins based on a specific condition ($\theta$) like <, >, =, etc.
    • Equi Join: A Theta Join where the condition only uses the equality operator (=).
    • Natural Join (⨝): An Equi Join where one of the two identical columns is automatically dropped from the result.
    • Outer Join: Keeps all tuples from one or both tables, even if there is no matching value in the other table. (Left Outer, Right Outer, Full Outer Join). Missing values are filled with NULL.

5. Relational Algebra Queries

Relational algebra operators are nested to form complex queries. The innermost operations are evaluated first.

Example Database:

Student(RollNo, Name, Age, DeptID)
Department(DeptID, DeptName)

  • Query 1: Find the names of all students who are older than 20.
    πName(σAge > 20(Student))
    Logic: First, filter the rows where Age > 20 using Selection (σ). Then, extract only the Name column using Projection (π).
  • Query 2: Find the names of students belonging to the 'Computer' department.
    πName(σDeptName = 'Computer'(Student ⨝ Department))
    Logic: First, Natural Join (⨝) Student and Department using common attribute DeptID. Then filter rows where DeptName is 'Computer' (σ). Finally, project the Name column (π).
  • Query 3: Find the names of all students who are either in 'IT' department OR are older than 22.
    πName(σDeptName = 'IT' ∨ Age > 22(Student ⨝ Department))
