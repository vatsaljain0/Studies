DBMS Module 4: Database Normalization

1. Pitfalls in Relational Design & Concept of Normalization

Normalization is the process of organizing data in a database to minimize redundancy and prevent data anomalies. Poorly designed (un-normalized) tables suffer from major pitfalls known as Anomalies.

The Three Anomalies

Bad Design Example: `Student_Course`

StudentIDStudentNameCourseIDInstructorName
101AliceCS1Dr. Smith
102BobCS1Dr. Smith
  • 1. Insertion Anomaly: Inability to add data to the database due to absence of other data.
    Example: We cannot add a new course (e.g., CS2 taught by Dr. Brown) until at least one student enrolls in it, because StudentID is the primary key and cannot be NULL.
  • 2. Deletion Anomaly: Unintentional loss of data due to deletion of other data.
    Example: If Alice and Bob both drop course CS1, we must delete their rows. Doing so completely erases the fact that Dr. Smith teaches CS1.
  • 3. Updation Anomaly: Data inconsistency resulting from data redundancy.
    Example: If Dr. Smith changes his name, we have to update multiple rows. If we miss one, the database becomes inconsistent.

2. Functional Dependencies & FD Closure

A Functional Dependency (FD) is a relationship that exists when one attribute uniquely determines another attribute.

Functional Dependency (FD) Definition
Notation X → Y
"X functionally determines Y", or "Y is functionally dependent on X"

If two rows share the exact same value for X, they MUST share the exact same value for Y.
Example: EmpID → EmpName. If two rows have EmpID = 5, their EmpName must be exactly the same.

Trivial vs. Non-Trivial FD

  • Trivial: X → Y is trivial if Y is a subset of X. (e.g., {EmpID, EmpName} → EmpID).
  • Non-Trivial: X → Y is non-trivial if Y is not a subset of X. (e.g., EmpID → EmpName).

FD Closure (Armstrong's Axioms)

The Closure of a set of FDs (denoted as F+) is the complete set of all FDs that can be logically deduced from the original set using Armstrong's Axioms.

Armstrong's Axioms & Attribute Closure
The 3 Primary Rules (Axioms) 1. Reflexivity: If Y ⊆ X, then X → Y
2. Augmentation: If X → Y, then XZ → YZ
3. Transitivity: If X → Y and Y → Z, then X → Z

Attribute Closure Example (X+)

Finding the closure of an attribute X (denoted X+) means finding all attributes that X can determine.

Given Relation R(A, B, C, D) and FDs: {A → B, B → C}
Find the closure of A (A+):
1. Start with the attribute itself: {A}
2. Since A → B, add B: {A, B}
3. Since B → C, add C: {A, B, C}
Result: A+ = {A, B, C}. (Because A cannot determine D, A is NOT a superkey).

The Normal Forms

Each normal form tackles a specific type of dependency to progressively eliminate redundancy.

1. First Normal Form (1NF)
Rule for 1NF A relation is in 1NF if and only if every attribute contains Atomic (indivisible) values. No multi-valued attributes or nested relations are allowed.

❌ BEFORE (Not in 1NF due to multi-valued Phone):

EmpIDNamePhoneNumbers
1John999-1234, 888-5678
✅ AFTER (Converted to 1NF by splitting rows):
EmpIDNamePhone
1John999-1234
1John888-5678
2. Second Normal Form (2NF)
Rule for 2NF 1. It must be in 1NF.
2. No Partial Dependencies. Every non-prime attribute (attributes not part of the Primary Key) must depend on the ENTIRE Primary Key, not just a part of it.

Scenario: Primary Key is composite: {StudentID, CourseID}.

❌ BEFORE (Not in 2NF):

StudentID (PK)CourseID (PK)CourseNameGrade
101C1PhysicsA
102C1PhysicsB

Problem: CourseName only depends on CourseID (a part of the PK). This is a Partial Dependency.

✅ AFTER (Decomposed into 2NF tables):

`Enrollment` Table

StudentIDCourseIDGrade
101C1A
102C1B

`Course` Table

CourseIDCourseName
C1Physics
3. Third Normal Form (3NF)
Rule for 3NF 1. It must be in 2NF.
2. No Transitive Dependencies. Non-prime attributes must depend ONLY on the Primary Key, and not on other non-prime attributes. (If A → B and B → C, then A → C is transitive).

Scenario: Primary Key is {EmpID}.

❌ BEFORE (Not in 3NF):

EmpID (PK)NameDeptIDDeptName
1JohnD1Sales

Problem: DeptName depends on DeptID, which depends on EmpID. This is a Transitive Dependency.

✅ AFTER (Decomposed into 3NF tables):

`Employee` Table

EmpIDNameDeptID
1JohnD1

`Department` Table

DeptIDDeptName
D1Sales
4. Boyce-Codd Normal Form (BCNF)
Rule for BCNF (Stricter 3NF) 1. It must be in 3NF.
2. For every non-trivial functional dependency X → Y, X must be a Super Key.

Scenario: A student can enroll in a course. A course can have multiple instructors, but an instructor teaches only ONE course.
Keys: {StudentID, Course} OR {StudentID, Instructor}.
FD: Instructor → Course.

❌ BEFORE (Not in BCNF):

StudentIDCourseInstructor
101JavaDr. A
102JavaDr. A

Problem: Instructor → Course, but 'Instructor' is NOT a super key (Dr. A can have multiple students). A non-prime attribute is determining a prime attribute.

✅ AFTER (Decomposed into BCNF tables):

`Student_Instructor` Table

StudentIDInstructor
101Dr. A
102Dr. A

`Instructor_Course` Table

InstructorCourse
Dr. AJava
5. Fourth Normal Form (4NF)
Rule for 4NF 1. It must be in BCNF.
2. No Multi-Valued Dependencies (MVDs). For a dependency X →→ Y, X must be a super key. You cannot have two independent multi-valued attributes in the same table.

Scenario: An employee can have multiple Skills and speak multiple Languages. Skills and Languages are completely independent of each other. (EmpID →→ Skill and EmpID →→ Language).

❌ BEFORE (Not in 4NF):

EmpIDSkillLanguage
1CodingEnglish
1CodingFrench
1DesignEnglish
1DesignFrench

Problem: Massive redundancy is generated to show all combinations of independent multi-valued facts.

✅ AFTER (Decomposed into 4NF tables):

`Emp_Skill` Table

EmpIDSkill
1Coding
1Design

`Emp_Language` Table

EmpIDLanguage
1English
1French
