DBMS Module 3: Structured Query Language (SQL)

1. Overview of SQL

  • What is SQL: Structured Query Language is used to manage relational databases. It allows users to create, modify, and retrieve data. It strongly follows the ACID (Atomicity, Consistency, Isolation, Durability) properties for transactions.
  • History: Developed by IBM (originally called Sequel) in the early 1970s as part of the System R project. It was standardized by ANSI in 1986.

Parts of the SQL Language:

  • Data-Definition Language (DDL): Defines the database structure using commands like CREATE, ALTER, and DROP.
  • Data-Manipulation Language (DML): Modifies database data using commands like SELECT, INSERT, UPDATE, and DELETE.
  • Integrity Constraints: DDL commands specify constraints that the data must satisfy (e.g., Primary Keys, Foreign Keys). Updates violating these are disallowed.
  • View Definition: DDL commands to define virtual tables (views).
  • Transaction Control Commands (TCL): Manages transactions using COMMIT and ROLLBACK.
  • Data Control Language (DCL): Manages permissions and security using GRANT and REVOKE.

2. Data Definition Commands (DDL) & Types

A. Built-in Data Types

  • char(n): Fixed-length character string.
  • varchar(n): Variable-length character string with max length n.
  • int / smallint: Integer and small integer types.
  • numeric(p, d): Fixed-point number with p total digits and d digits to the right of the decimal (e.g., numeric(3,1) handles exactly 44.5).
  • real / double precision / float(n): Floating-point numbers.
  • BLOB: Binary Large Object used for storing large binary values like images.
  • DATE / TIME / TIMESTAMP: DATE handles YYYY-MM-DD; TIME handles HH:MM:SS; TIMESTAMP includes both plus fractions of seconds.

B. Schema & Constraints Definition

  • The CREATE TABLE command specifies a new relation by giving it a name, its attributes, and initial constraints.
  • Integrity Constraints: Key constraints (Primary Key), entity constraints (NOT NULL), and referential integrity (Foreign Key references) can be declared within the CREATE statement or added later via ALTER TABLE.
DDL Example: Creating Tables & Constraints

Syntax Rule: Define attributes, data types, and integrity constraints (Primary Key, Foreign Key).

create table department ( dept_name varchar(20), building varchar(15), budget numeric(12,2), primary key (dept_name) ); create table instructor ( ID varchar(5), name varchar(20) not null, dept_name varchar(20), salary numeric(8,2), primary key (ID), foreign key (dept_name) references department );
Output Result

Empty schema tables for department and instructor are created in the database, enforcing that an instructor's dept_name MUST exist in the department table (Referential Integrity).

3. Data Manipulation Commands (DML)

A. Single Relation Operations

  • INSERT: Adds new tuples (rows) to a relation.
  • DELETE: Removes tuples from a relation.
  • ALTER / DROP: Modifies the schema of a relation or removes the table entirely.
  • SELECT: Used to list desired attributes. The distinct keyword removes duplicates from the result set.

B. Multiple Relations & Cartesian Product

An SQL query primarily contains three main clauses: select, from, and where. Execution logic operates in this specific order:

  1. The from clause by itself generates a Cartesian product of all the relations listed.
  2. The where clause predicate is then applied to restrict the combinations created by the Cartesian product to meaningful, related tuples.
  3. The select clause outputs the required attributes from the restricted tuples.

4. Rename, String Operations & Ordering

A. The Rename Operation

SQL uses the as clause (old-name as new-name) in both select and from clauses to rename relations or attributes. It is useful for creating convenient shorthands (e.g., from instructor as T).

B. String Operations

  • Strings are enclosed in single quotes and are case-sensitive (e.g., 'Comp. Sci.').
  • Pattern matching uses the like or not like operators.
  • Percent (%): Matches any substring.
  • Underscore (_): Matches exactly one single character.
  • ESCAPE: Defines a character used immediately before a special pattern character to treat it as a normal character (e.g., like 'ab\%cd%' escape '\').

C. Ordering Tuples

The order by clause causes the result tuples to appear in sorted order. Use desc for descending and asc for ascending order.

5. Set Operations & Null Values

A. Set Operations

SQL operations correspond directly to mathematical set theory:

  • UNION: Combines results. By default, it automatically eliminates duplicates. Use union all to retain duplicates.
  • INTERSECT: Returns tuples present in both queries. Eliminates duplicates unless intersect all is used.
  • EXCEPT: Returns tuples from the first query that are absent in the second. Eliminates duplicates unless except all is used.

B. Null Values

  • Arithmetic operations (+, -, *, /) involving a null value always result in a null.
  • Comparisons involving nulls (e.g., "1 < null") are treated as unknown.
  • Boolean operations are extended to handle unknowns (e.g., true or unknown is true; true and unknown is unknown).
  • To test for nulls, SQL uses the explicit keyword is null in a predicate.
Set Operations (UNION, INTERSECT, EXCEPT)

Sample Data from `section` table:

  • Spring 2010 Courses: CS-101, CS-315, FIN-201
-- Find courses taught in Fall 2009 BUT NOT in Spring 2010 (select course_id from section where semester = 'Fall' and year = 2009) except (select course_id from section where semester = 'Spring' and year = 2010);

If we used intersect, the output would be only CS-101 (present in both).

6. Aggregate Functions, Group By & Having

Aggregate functions take a collection of values and return a single value. The five built-in functions are: avg, min, max, sum, and count.

GROUP BY & HAVING

  • The group by clause places tuples satisfying the where predicate into groups.
  • The having clause states conditions that apply to groups rather than individual tuples. Predicates in the having clause are applied after groups have been formed.
  • Crucial Rule: Any attribute present in the having clause without being aggregated must appear in the group by clause.
Aggregates with GROUP BY and HAVING

7. Nested Subqueries

A subquery is a select-from-where expression nested within another query. It is used to perform tests for set membership, comparisons, or cardinality.

  • Set Membership (IN / NOT IN): The in connective tests if a value exists within the collection produced by a subquery.
  • Set Comparison (SOME / ALL): The phrase "greater than at least one" is represented by > some. The construct > all translates to "greater than all" values in the subquery.
  • Test for Empty Relations (EXISTS): The exists construct returns true if the subquery contains any tuples (is nonempty). If it utilizes a correlation name from the outer query, it's called a correlated subquery.
  • Test for Duplicates (UNIQUE): The unique construct returns true if the subquery contains absolutely no duplicate tuples.
  • The WITH Clause: Defines a temporary relation whose definition is only available to the current query.
Set Comparison: SOME and ALL

Sample Data Context:

Assume the Biology department has instructors with salaries: 72000 and 80000.

-- Find names of all instructors whose salary is greater than AT LEAST ONE instructor in Biology select name from instructor where salary > some (select salary from instructor where dept_name = 'Biology');
Output Logic

Returns any instructor from the database making more than 72000 (because 72000 is the lowest value in the Biology subquery, satisfying "> some").

-- Find names of all instructors whose salary is greater than ALL instructors in Biology select name from instructor where salary > all (select salary from instructor where dept_name = 'Biology');
Output Logic

Returns only instructors making strictly more than 80000 (must be greater than every single value returned by the subquery).

The WITH Clause (Temporary Views)
-- Find those departments with the maximum budget with max_budget (value) as (select max(budget) from department) select budget from department, max_budget where department.budget = max_budget.value;

8. SQL Joins

A JOIN clause is used to combine rows from two or more tables based on a related column between them. The on condition allows a general predicate to govern the join.

EQUI JOIN uses the equal sign (=), while NON EQUI JOIN uses operators like >, <, >=, or <=.

Types of Joins:

  • INNER JOIN: Returns only the rows that have matching values in both participating tables.
  • LEFT (OUTER) JOIN: Returns all records from the left table, and the matched records from the right table. Unmatched rows are populated with null values.
  • RIGHT (OUTER) JOIN: Returns all records from the right table, and the matched records from the left table.
  • FULL (OUTER) JOIN: Returns all records when there is a match in either the left or the right table, combining both left and right outer join results.
  • NATURAL JOIN: Considers only those pairs of tuples with the exact same value on identically named attributes. The common columns appear only once in the result.
  • CROSS JOIN: Produces a Cartesian Product by multiplying the rows of the first table with the second table.
  • SELF JOIN: A regular join where a table is joined with itself.

9. Views, Triggers & PL/SQL

A. Views in SQL

A view acts as a virtual relation. The database system only stores the query expression, calculating the result whenever accessed.

  • Materialized Views: Some databases allow view relations to be actually stored on disk. The system ensures that if the underlying relations change, the view is automatically kept up-to-date.
  • Updatable Views: Updates, inserts, or deletes can be applied directly to a view only if it satisfies strict conditions: The FROM clause has only one relation, no aggregate/DISTINCT expressions, no GROUP BY/HAVING clauses, and unlisted attributes can be safely set to null.
Creating a View

A view is a virtual table whose contents are defined by a query. It provides data abstraction.

-- Hide salary data from a clerk by providing a specific view create view faculty as select ID, name, dept_name from instructor; -- You can query a view exactly like a real table select * from faculty;

B. Triggers

  • Mechanism: Requires specifying when the trigger executes (the event and condition) and the actions to be taken.
  • Need: Used to implement complex integrity constraints that cannot be handled by standard SQL constraints. Useful for alerting humans or automatically starting tasks.
  • When NOT to use Triggers: Modern databases handle materialized view maintenance and replication automatically, making triggers unnecessary for these. Triggers can also cause massive issues via unintended execution when data is loaded from a backup copy.
Creating a Trigger

A trigger executes automatically as a side effect of a modification (Event + Condition → Action).

-- Maintain credits earned values when a grade is updated create trigger credits_earned after update of takes on (grade) referencing new row as nrow referencing old row as orow for each row when nrow.grade <> 'F' and nrow.grade is not null and (orow.grade = 'F' or orow.grade is null) begin atomic update student set tot_cred = tot_cred + (select credits from course where course.course_id = nrow.course_id) where student.id = nrow.id; end;

C. PL/SQL Block Structure

PL/SQL (Procedural Language extension to SQL) allows writing procedural code that executes within the Database. The fundamental unit is a Block.

PL/SQL Block Syntax
DECLARE -- (Optional) Declare variables, constants, cursors v_name varchar(20); BEGIN -- (Mandatory) Executable statements & core logic SELECT name INTO v_name FROM instructor WHERE ID = '10101'; EXCEPTION -- (Optional) Error handling to prevent crashes WHEN NO_DATA_FOUND THEN DBMS_OUTPUT.PUT_LINE('No such instructor found'); END; -- (Mandatory) Marks conclusion
