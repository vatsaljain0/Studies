Deconstructed Syntax, Detailed Theory & Isolated Examples

1. Introduction and Overview of SQL

SQL Basics & Command Categories

SQL is used to manage relational databases by creating, modifying, and retrieving data. It strictly follows ACID (Atomicity, Consistency, Isolation, Durability) properties for transactions.

  • Data Definition Language (DDL): Defines the database structure (schema, constraints, indices) using commands like CREATE, ALTER, and DROP.
  • Data Manipulation Language (DML): Modifies and retrieves database data using SELECT, INSERT, UPDATE, and DELETE.
  • Transaction Control Language (TCL): Manages transactions using COMMIT and ROLLBACK.
  • Data Control Language (DCL): Manages permissions using GRANT and REVOKE.

2. SQL Data Types

Built-in Data Types

SQL standard supports a variety of built-in types for defining attributes:

  • Character: char(n) (fixed-length, pads with spaces) and varchar(n) (variable-length, up to max length n).
  • Numeric: int, smallint, numeric(p, d) (fixed-point precision where p=total digits, d=decimal places), real, double precision, and float(n).
  • Binary & Bit Strings: BIT(n), BIT VARYING(n), and BLOB (Binary Large Object, used for storing large binary values like images).
  • Boolean: TRUE, FALSE, or UNKNOWN (Three-valued logic is used because of Null values).
  • Date & Time: DATE (YYYY-MM-DD), TIME (HH:MM:SS), TIME WITH TIME ZONE, and TIMESTAMP (combines date, time, and decimal fractions of seconds).

3. Basic Schema Definition & Integrity Constraints

Concept: Integrity Constraints

Integrity constraints are strict rules applied to columns to ensure accuracy, reliability, and validity of data. Any INSERT, UPDATE, or DELETE operation violating these rules is completely rejected by the database.

  • 1. Domain Constraints: Dictate exactly what kind of data is permitted in a given attribute. This involves setting a Data Type and modifiers like NOT NULL (which prevents an attribute from being left empty or undefined).
  • 2. Key Constraints (Entity Integrity): Ensures every row can be uniquely identified. A PRIMARY KEY enforces that values are 100% unique across the table and inherently never null.
  • 3. Referential Integrity (Foreign Key Constraints): Maintains valid relationships across tables. By using a FOREIGN KEY, SQL ensures that a value appearing in the child table must already exist in the referenced parent table, preventing "orphan" records.
  • 4. Check Constraints: Evaluates a specific custom Boolean condition before allowing data insertion/updation (e.g., ensuring a salary value is > 0, or age >= 18).
CREATE, ALTER, DROP & Applying Constraints
Generic Syntax
CREATE TABLE table_name ( col datatype constraint ); ALTER TABLE table_name ADD col datatype; DROP TABLE table_name;
  • CREATE TABLE: Allocates space for a new schema.
  • constraint: Keywords like PRIMARY KEY, NOT NULL, or CHECK(...).
  • FOREIGN KEY (col) REFERENCES other_table(col): The syntax to create a Referential Integrity link.
  • ALTER / DROP: Adds/Removes attributes, or completely destroys the table.
Execution Example
-- Creating tables and applying all 4 constraint types CREATE TABLE Department ( dept_id INT PRIMARY KEY, -- Key Constraint dept_name VARCHAR(50) NOT NULL -- Domain Constraint ); CREATE TABLE Employee ( emp_id INT PRIMARY KEY, name VARCHAR(50), salary DECIMAL(10,2) CHECK (salary > 0), -- Check Constraint dept_id INT, FOREIGN KEY (dept_id) REFERENCES Department(dept_id) -- Referential Integrity ); -- Modifying Tables ALTER TABLE Employee ADD age INT; ALTER TABLE Employee DROP COLUMN age; DROP TABLE Employee;

SELECT (Retrieval & Filtering)
Generic Syntax
SELECT [DISTINCT | ALL | *] col1 FROM table_name WHERE condition;
  • SELECT: Command to fetch data.
  • *: Symbol to select all attributes (columns).
  • DISTINCT: Removes duplicate rows from the output.
  • ALL: Retains duplicates (this is the default behavior if omitted).
Execution Example

TCL & DCL Commands
Generic Syntax
COMMIT; | ROLLBACK; GRANT privileges ON table TO user; | REVOKE privileges ON table FROM user;
  • COMMIT: Permanently saves transactions. ROLLBACK undoes them.
  • GRANT / REVOKE: Manages permissions.
Execution Example

5. Rename, String Operations & Ordering

Rename (AS), Strings & Ordering
Generic Syntax
SELECT col AS new_name FROM table WHERE col LIKE 'pattern' ESCAPE '\' ORDER BY col DESC;
  • AS: Renames attributes or relations in the output (e.g., old-name AS new-name).
  • ||, UPPER(), LOWER(), TRIM(): String functions for concatenation, case changing, and whitespace removal.
  • LIKE / NOT LIKE: Pattern matching. % matches any substring. _ matches exactly one single character.
  • ESCAPE '\': Allows you to search for literal % or _ characters by placing a \ in front of them.
  • ORDER BY ... ASC/DESC: Sorts results ascending or descending.
  • BETWEEN: Used in the WHERE clause instead of >= AND <=.
Execution Example

6. Set Operations

UNION, INTERSECT, EXCEPT
Generic Syntax
SELECT col FROM A [UNION | INTERSECT | EXCEPT] SELECT col FROM B;
  • UNION: Combines the results of both queries. Automatically eliminates duplicates. (Use UNION ALL to retain duplicates).
  • INTERSECT: Returns only rows present in BOTH queries. Eliminates duplicates (unless INTERSECT ALL is used).
  • EXCEPT: Returns rows in the first query that are NOT in the second query. Eliminates duplicates (unless EXCEPT ALL is used).
Execution Example

Assuming Table A has [10, 20] and Table B has [20, 30].

SELECT dept_id FROM A EXCEPT SELECT dept_id FROM B;
Output Result:

10

7. Null Values

Handling Nulls & Three-Valued Logic

Null values present special problems in relational operations:

  • Arithmetic: Any arithmetic operation (+, -, *, /) involving a NULL results in a NULL (e.g., 5 + null = null).
  • Comparisons: Comparisons involving nulls (like 1 < null) are treated as unknown, not True or False.
  • Three-Valued Logic: Because of unknowns, Boolean logic is expanded:
    • AND: True AND Unknown = Unknown. False AND Unknown = False.
    • OR: True OR Unknown = True. False OR Unknown = Unknown.
    • NOT: NOT Unknown = Unknown.

8. Aggregate Functions & Grouping

avg, min, max, sum, count & GROUP BY
Generic Syntax
SELECT col1, AVG(col2) FROM table GROUP BY col1 HAVING AVG(col2) > val;
  • AVG, MIN, MAX, SUM, COUNT: Built-in aggregate functions that take a collection of values and return a single value.
  • GROUP BY: Partitions tuples into groups before applying the aggregate function.
  • HAVING: Applies predicates to groups after they have been formed (unlike WHERE, which filters individual tuples before grouping).
Execution Example

9. Nested Subqueries

Subqueries are select-from-where expressions nested inside another query. They are used for set membership, comparisons, or cardinality checks.

IN, > SOME, > ALL, EXISTS, UNIQUE, WITH
Operators Breakdown
  • IN / NOT IN (Set Membership): Tests if a value exists within the collection produced by the subquery.
  • > SOME / = SOME (Set Comparison): "Greater than at least one". (Note: = SOME is identical to IN).
  • > ALL / = ALL (Set Comparison): "Greater than all". (Note: = ALL is identical to NOT IN).
  • EXISTS / NOT EXISTS (Empty Relation): Returns true if the subquery returns ANY tuples. A subquery that uses a correlation name from an outer query is called a correlated subquery.
  • UNIQUE / NOT UNIQUE (Duplicate Tuples): Returns true if the subquery contains absolutely no duplicate tuples.
  • WITH Clause: Defines a temporary relation whose definition is available only to the current query.
Execution Examples
-- 1. SET MEMBERSHIP (IN) SELECT name FROM Employee WHERE dept_id IN (SELECT dept_id FROM Department WHERE dept_name = 'IT'); -- 2. SET COMPARISON (> ALL) -- Find employees whose salary is greater than EVERY employee in dept 10 SELECT name FROM Employee WHERE salary > ALL (SELECT salary FROM Employee WHERE dept_id = 10); -- 3. THE WITH CLAUSE & SUBQUERY IN FROM CLAUSE -- Create a temporary relation 'MaxSal' using WITH, then join it WITH MaxSal(val) AS (SELECT MAX(salary) FROM Employee) SELECT name FROM Employee, MaxSal WHERE Employee.salary = MaxSal.val;

10. SQL Joins

INNER, OUTER, NATURAL, CROSS, SELF
Join Types Breakdown
  • CROSS JOIN (Cartesian Product): Multiplies every row of table 1 with table 2. Implicitly generated by the FROM clause if multiple tables are listed.
  • INNER JOIN (Equi Join): Returns ONLY matched rows between tables using the ON condition.
  • LEFT OUTER JOIN: Returns all records from the left table + matched records from the right. Unmatched are set to NULL.
  • RIGHT OUTER JOIN: Returns all records from the right table + matched records from the left.
  • FULL OUTER JOIN: Returns ALL matched and unmatched rows from both tables.
  • NATURAL JOIN: An Equi Join that automatically equates pairs of identically named columns and eliminates redundant columns from the output.
  • SELF JOIN: Joining a table with itself, typically when a foreign key references its own primary key.

11. SQL Views (5-6 Marks Short Note)

What is a View?

A View is a virtual table whose contents are defined by a query. Unlike a regular table, a standard view does not store data physically on the disk; the database only stores the underlying SELECT statement and evaluates it dynamically every time the view is accessed.

Advantages / Why use Views?

  • Security: You can hide sensitive columns (like salary) or rows by providing users access only to the View, not the base tables.
  • Simplicity (Data Abstraction): Complex queries containing multiple JOINs can be saved as a single View, making it easier for end-users to query.
  • Logical Data Independence: If the underlying table structure changes, the view can be modified to keep the user-facing schema identical.

Types of Views

  • Standard/Virtual Views: Evaluated on-the-fly. Do not consume disk space for data.
  • Materialized Views: The result of the query is actually physically stored on the disk. They are used in data warehousing to speed up extremely complex, slow queries. The database must periodically refresh this stored data to keep it consistent.
  • Updatable Views: You can perform INSERT, UPDATE, or DELETE directly on a view only if it meets strict rules:
    • The view is based on a single table (no JOINs).
    • No aggregate functions (SUM, AVG) or DISTINCT keyword.
    • No GROUP BY or HAVING clauses.
CREATE VIEW Example
-- Hide salary data from public users CREATE VIEW Public_Employee_Info AS SELECT emp_id, name, dept_id FROM Employee; -- Using the view just like a real table SELECT * FROM Public_Employee_Info;

12. Triggers (5-6 Marks Short Note)

What is a Trigger?

A Trigger is a special type of stored procedure that is automatically executed (or "fired") by the DBMS as a side effect of a specific database modification event (such as INSERT, UPDATE, or DELETE) on a particular table.

ECA Architecture of a Trigger

Triggers strictly follow the ECA (Event-Condition-Action) rule:

  • Event: The database operation that causes the trigger to fire (e.g., AFTER INSERT or BEFORE DELETE).
  • Condition (Optional): A boolean rule evaluated when the event occurs (e.g., WHEN new.salary > 100000). The action only executes if this is true.
  • Action: The PL/SQL procedural code executed by the trigger to perform the intended task.

Applications / Why use Triggers?

  • Enforcing Complex Business Rules: When a simple CHECK constraint isn't enough (e.g., checking values across multiple different tables before allowing an insert).
  • Auditing and Logging: Automatically creating a log record of who modified a row, when they did it, and what the old values were.
  • Automatic Derived Calculations: (e.g., Automatically subtracting stock quantity in the Inventory table whenever a new row is inserted into the Orders table).
CREATE TRIGGER Example
Structure
CREATE TRIGGER name [BEFORE | AFTER] [INSERT | UPDATE | DELETE] ON [table] REFERENCING NEW ROW AS nrow FOR EACH ROW WHEN [condition] BEGIN ATOMIC ... END;
Execution Example: Auto-Updating Inventory
-- Automatically update inventory when a new order is inserted CREATE TRIGGER Update_Inventory AFTER INSERT ON Orders REFERENCING NEW ROW AS nrow FOR EACH ROW BEGIN ATOMIC UPDATE Inventory SET quantity = quantity - nrow.amount WHERE item_id = nrow.item_id; END;

13. Introduction to PL/SQL Block Structure

Procedural Extensions

PL/SQL wraps standard SQL queries inside programming blocks, allowing for variables, IF/ELSE statements, loops, and error handling. The fundamental unit is a Block.

PL/SQL Block
Generic Syntax
DECLARE ... BEGIN ... EXCEPTION ... END;
  • DECLARE: (Optional) Used to declare variables, constants, or cursors.
  • BEGIN: (Mandatory) The start of the actual executable code and core logic.
  • EXCEPTION: (Optional) Used to catch and handle errors (e.g., if a query returns zero rows).
  • END;: (Mandatory) Marks the conclusion of the block.
Execution Example
DECLARE v_name VARCHAR(50); BEGIN SELECT name INTO v_name FROM Employee WHERE emp_id = 1; IF v_name = 'John' THEN DBMS_OUTPUT.PUT_LINE('Hello John!'); END IF; EXCEPTION WHEN NO_DATA_FOUND THEN DBMS_OUTPUT.PUT_LINE('Error: Employee does not exist!'); END;
