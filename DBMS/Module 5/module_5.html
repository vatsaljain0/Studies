1. Transaction Basics & ACID

What is a Transaction?

Simple version: A single logical task, like a bank transfer, even if it has many steps.

Exam Version (Proper Definition)
A Transaction is a unit of program execution that accesses and possibly updates various data items. It is considered the smallest logical unit of processing in a database.
The ACID Properties

Simple version: All or Nothing, No Rules Broken, No Peeking, Permanent.

Exam Version (Formal Definitions)
  • Atomicity: Ensures that all operations of a transaction are completed successfully, or none are. If a failure occurs, the database is rolled back to its previous state.
  • Consistency: Ensures that a transaction transforms the database from one valid state to another, maintaining all predefined integrity constraints.
  • Isolation: Ensures that the execution of concurrent transactions does not interfere with each other; intermediate results remain invisible to other transactions.
  • Durability: Ensures that once a transaction is committed, its changes are permanent and survive any subsequent system failures.

2. Serializability & Precedence Graphs

S (Actual Schedule): The interleaved execution we are testing.

S' (Serial Schedule): The reference where transactions run one-after-another.

Conflict Serializability

Simple version: If you can rearrange a messy schedule into a serial one without breaking data logic, it's safe.

Exam Version (The Formal Logic)
A schedule S is Conflict Serializable if it is conflict-equivalent to some serial schedule S'. Two operations are in Conflict if:
  • They belong to different transactions.
  • They access the same data item Q.
  • At least one of them is a Write(Q).
Precedence Graph Example (Solving Numericals)

In exams, you'll get a table of operations. Here is how you prove it's serializable:

Schedule S:
1. T1: Read(A)
2. T2: Read(A)
3. T1: Write(A)
4. T2: Write(A)

Step-by-Step Logic:

  1. Check Step 1 & 4: T1 reads A, T2 later writes A. Conflict! Draw arrow T1 → T2.
  2. Check Step 2 & 3: T2 reads A, T1 later writes A. Conflict! Draw arrow T2 → T1.

Result: Since we have arrows going both ways (T1 ↔ T2), a Cycle exists. This schedule is NOT Conflict Serializable.

View Serializability

Simple version: Only cares if the final "view" of the data is the same as the serial version.

Exam Version (The 3 Conditions)
A schedule S is View Serializable if it is view-equivalent to a serial schedule S'. This requires:
  1. Initial Read: If T1 reads initial data in S, it must do so in S'.
  2. Read-From: If T1 reads data written by T2 in S, it must do so in S'.
  3. Final Write: Whoever updates a data item last in S must also be the last in S'.

3. Lock-Based Protocols (2PL)

Lock Modes

Shared (S) = Read only. Exclusive (X) = Read & Write.

Exam Version (Formal Definitions)
  • Shared Mode (S): Allows a transaction to read a data item but not update it. Multiple transactions can hold S-locks on the same item.
  • Exclusive Mode (X): Allows a transaction to both read and update an item. It is exclusive; no other transaction can hold any lock on the same item.

The 3 Variants of 2PL

Variant Exam-Ready Rule Standard Benefit
Basic 2PL Transaction acquires all locks in Growing phase, releases in Shrinking phase. Ensures Conflict Serializability.
Strict 2PL Holds all Exclusive (X) locks until Commit/Abort. Prevents Cascading Rollbacks.
Rigorous 2PL Holds ALL locks (S and X) until Commit/Abort. Safest, ensures strict serializability.

4. Deadlock Handling

What is a Deadlock?

Simple version: A "Mexican Standoff" where Task A waits for Task B, and Task B waits for Task A. No one can move.

Exam Version (Technical Definition)
A Deadlock is a state in which every transaction in a set of transactions is waiting for an event (usually a lock release) that can only be caused by another transaction in that same set. This is known as a Circular Wait.

Handling Strategies

1. Deadlock Detection

The system maintains a Wait-for Graph. If a cycle is detected, the system picks a "victim" and rolls it back.

2. Deadlock Prevention

Uses Wait-Die (older waits) or Wound-Wait (older kicks out younger) schemes based on timestamps.

5. Log-Based Recovery

The Recovery Process

The system keeps a "Diary" (Log). When a crash occurs, it follows these two rules:

  • REDO: If a transaction Committed before the crash, re-apply its changes from the log.
  • UNDO: If a transaction was still Active during the crash, reverse its changes to keep DB clean.
Exam Version (Deferred vs Immediate)
Method Mechanism Recovery Action
Deferred Update Postpones updates to the actual database until the transaction commits. REDO only.
Immediate Update Updates the database directly as operations occur (while transaction is active). Both UNDO and REDO.

Checkpoint: A specialized log record indicating that all previous updates were successfully written to the physical disk. It minimizes recovery time by limiting the log search to records after the checkpoint.
