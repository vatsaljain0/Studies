Transactions & Concurrency Control

Comprehensive Exam Guide: Protocols, Deadlocks, and Recovery

1. Transaction Concept & ACID Properties

What is a Transaction?

A transaction is a single, logical unit of work that accesses and possibly updates various data items in a database. A transaction must either execute completely or not at all.

Classic Example: Fund Transfer

Transferring $50 from Account A to Account B involves several steps:
1. Read(A) → 2. A = A - 50 → 3. Write(A) → 4. Read(B) → 5. B = B + 50 → 6. Write(B).
If the system crashes after step 3, $50 is deducted from A but never added to B. The money vanishes! This is why transaction protocols are vital.

The ACID Properties

To maintain data integrity during concurrent executions and system failures, the DBMS guarantees four properties:

  • Atomicity (All-or-Nothing): Ensures that either all operations of the transaction reflect properly in the database, or none do. There are no partial transactions.
  • Consistency: Execution of a transaction in isolation preserves the consistency of the database (e.g., the sum of Account A and B must be the same before and after the transfer).
  • Isolation: Even though multiple transactions execute concurrently, the system guarantees that each transaction is unaware of the others. A transaction's intermediate (unfinished) changes remain invisible to others.
  • Durability: Once a transaction successfully completes (commits), its changes are permanent and survive any future system crashes.

2. Transaction States & Concurrent Executions

The Transaction State Machine
  • Active: The initial state. The transaction stays here while executing read/write operations.
  • Partially Committed: Reached after the final statement executes. The data is still in the memory buffer and hasn't been permanently saved to the disk yet.
  • Committed: Reached after successful completion and data is safely stored on the physical disk.
  • Failed: Occurs when the system discovers that normal execution can no longer proceed (e.g., due to software logic error or hardware crash).
  • Aborted: The transaction has been rolled back and the database is restored to its prior state. The OS can either restart the transaction or kill it.

Concurrent Executions & Why We Do It

A system allows multiple transactions to execute simultaneously. We do this to improve Throughput (transactions completed per second) and reduce Response Time (users don't have to wait in a single-file line).

The 3 Race Conditions (Why concurrency goes wrong):

  • Lost Update: ATM-1 reads balance ₹10,000. ATM-2 simultaneously reads ₹10,000. Both deduct ₹8,000 and write it back. One update overwrites the other, and the bank loses track of ₹8,000!
  • Dirty Read: T2 reads data that T1 has written, before T1 commits. If T1 crashes and rolls back, T2 has processed a calculation using "dirty" (invalid) data.
  • Unrepeatable Read: T1 reads a value X as 100. T2 updates X to 200 and commits. If T1 reads X again in the same transaction, it gets 200. The data magically changed mid-transaction!

3. Serializability (Conflict & View)

Schedules and Serializability

A Schedule is a chronological sequence of instructions indicating the order in which transactions execute. A Serial Schedule executes transactions one-by-one without any overlap (perfectly safe, but horribly slow).

Serializability is the process of taking a fast, concurrent schedule and mathematically proving that it yields the exact same outcome as a safe, serial schedule.

Conflict Serializability

Two operations from different transactions are said to Conflict if they access the same data item and at least one of them is a Write operation.

  • Read(Q) & Read(Q) ➔ No Conflict (Safe to swap order)
  • Read(Q) & Write(Q) ➔ Conflict! (Order matters)
  • Write(Q) & Write(Q) ➔ Conflict! (Order matters)

If a concurrent schedule can be transformed into a serial schedule by swapping non-conflicting operations, it is Conflict Serializable.

The Precedence Graph Algorithm (Exam Proof)

To prove conflict serializability in an exam, draw a Precedence Graph. Create a node for every transaction. Draw an arrow from T1 to T2 if T1 executes a conflicting operation before T2.
The Rule: If the graph contains a Cycle (a circle), the schedule is NOT serializable. If there is no cycle, it is serializable.

View Serializability

A schedule is View Serializable if it produces the exact same "view" (read values and final written values) as a serial schedule.

All conflict-serializable schedules are automatically view-serializable. However, some schedules are view-serializable but NOT conflict-serializable! This happens because of Blind Writes (when a transaction writes a value without ever reading it first, masking previous conflicts).

4. Concurrency Control: Lock-Based Protocols & 2PL

Lock-Based Protocols

To prevent Race Conditions, a transaction must obtain a "lock" on a data item before accessing it.

  • Shared Lock (S-Lock): Allows the transaction to only Read the data. Multiple transactions can hold S-Locks on the same data simultaneously.
  • Exclusive Lock (X-Lock): Allows the transaction to both Read and Write. If T1 holds an X-Lock, NO other transaction can get any lock (S or X) on that item. They must wait.
Two-Phase Locking Protocol (2PL)

The 2PL protocol guarantees conflict serializability by demanding that locking happens in two strict phases:

  • 1. Growing Phase: A transaction may acquire new locks, but it is strictly forbidden from releasing any locks.
  • Lock Point: The exact moment the transaction has acquired all the locks it will ever need.
  • 2. Shrinking Phase: A transaction may release locks, but it is strictly forbidden from acquiring any new locks.

The Problem:

While 2PL prevents bad schedules, it does not prevent Deadlocks. It also allows Cascading Rollbacks (if T1 fails, T2 which read T1's unlocked data must also be rolled back).

The 3 Variants of 2PL

Basic 2PLStrict 2PLRigorous 2PL
Transaction acquires locks in the Growing phase and releases them in the Shrinking phase. Ensures conflict serializability but allows cascading rollbacks. Holds all Exclusive (X) locks until the transaction commits or aborts. (S-Locks can be released early). Prevents cascading rollbacks. Holds ALL locks (both S and X) until the transaction commits or aborts. Safest variant, prevents cascading rollbacks, but reduces concurrency.

5. Introduction to Deadlock Handling

What is a Deadlock?

A deadlock occurs when a set of transactions are locked in a Circular Wait. For example: Transaction 1 holds a lock on Data A and is waiting for Data B. Transaction 2 holds a lock on Data B and is waiting for Data A. Neither can proceed. They are frozen forever.

Deadlock Detection & Prevention

1. Deadlock Detection (Wait-For Graph)

The OS builds a directed graph where nodes are transactions. An arrow from T1 to T2 means T1 is waiting for a lock held by T2. If the OS detects a cycle/loop in this graph, a deadlock has occurred.

2. Deadlock Prevention Algorithms

  • Timeout: A transaction waits for a lock. If the clock runs out, it assumes a deadlock occurred and just rolls itself back.
  • Wound-Wait (Preemptive): "Older wounds younger." If Old-T1 wants a lock held by Young-T2, T1 wounds (aborts) T2 to take the lock. If Young-T2 wants a lock from Old-T1, T2 waits.
  • Wait-Die (Non-Preemptive): "Younger dies." If Old-T1 wants a lock held by Young-T2, T1 waits. If Young-T2 wants a lock from Old-T1, T2 dies (aborts itself).

3. Deadlock Recovery Steps

  1. Select a Victim: Choose the transaction that will cost the least to kill (fewest updates made).
  2. Abort / Rollback: Undo all the changes the victim made to the database.
  3. Release: Release all locks held by the victim so others can proceed.
  4. Retry: Let the surviving transactions finish, then restart the victim later.

6. Concurrency Control: Timestamp-Based Protocols

Lock-Free Concurrency

Instead of using locks (which cause deadlocks), we use time. The DBMS gives every transaction a unique Timestamp TS(Ti) when it begins. An older transaction has a smaller timestamp. We force transactions to execute as if they ran sequentially in timestamp order.

Every data item (Q) stores two markers:

  • W-timestamp(Q): The largest timestamp of any transaction that successfully wrote to Q.
  • R-timestamp(Q): The largest timestamp of any transaction that successfully read Q.
Timestamp Ordering Rules

If Transaction Ti wants to READ(Q):

  • If TS(Ti) < W-timestamp(Q), this means a younger transaction already overwrote the data Ti was supposed to read! Ti is reading from the future. Ti is rolled back.
  • Otherwise, the read succeeds, and R-timestamp(Q) is updated.

If Transaction Ti wants to WRITE(Q):

  • If TS(Ti) < R-timestamp(Q), a younger transaction already read the old value. If Ti writes now, the younger one's read is invalid. Ti is rolled back.
  • If TS(Ti) < W-timestamp(Q), a younger transaction already wrote over this data. Ti is rolled back.
  • Otherwise, the write succeeds, and W-timestamp(Q) is updated.

Thomas' Write Rule (Optimization)

A brilliant modification to the protocol. If TS(Ti) < W-timestamp(Q), Ti is trying to write an obsolete (outdated) value. Instead of harshly rolling back Ti and killing it, Thomas' rule says: Just ignore the write operation and let Ti continue! This prevents unnecessary aborts and increases concurrency.

7. Recovery System: Log-Based Recovery

Write-Ahead Logging (WAL)

To ensure Atomicity and Durability, the DBMS keeps a diary of everything it does, called a System Log. The golden rule is Write-Ahead Logging (WAL): A log record detailing a change MUST be physically written to the hard disk before the actual database is updated.

Log Records contain:

  • Transaction ID (Which transaction did this?)
  • Data Item (What was modified?)
  • Old Value (What was the value before? Used for UNDO).
  • New Value (What is the value now? Used for REDO).
Recovery Operations: UNDO and REDO

When a system crashes and reboots, the Recovery Manager reads the log from top to bottom and performs two actions to fix the database:

  • UNDO: Restores the old values. Applied to any transaction that has a <T_start> log entry but is missing a <T_commit> entry. Because it crashed midway, we must undo its partial changes to maintain Atomicity.
  • REDO: Re-applies the new values. Applied to any transaction that successfully recorded a <T_commit> entry before the crash. This guarantees Durability (even if the actual database memory buffer was destroyed, the log remembers what the new values were supposed to be).

Checkpoints

If the DB runs for a year and crashes, reading a 1-year-long log file during reboot would take days! A Checkpoint is a mechanism where the DBMS periodically pauses, forces all modified memory buffers to the physical disk, and writes a <checkpoint> marker in the log. During a crash, the recovery manager only needs to read the log backwards until it hits the last checkpoint, ignoring the months of data before it.
