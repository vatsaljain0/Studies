Computational Theory: Decidability and Computability

1. Recursive and Recursively Enumerable Languages

The concepts of Recursive (REC) and Recursively Enumerable (RE) languages classify how Turing Machines (TMs) interact with inputs.

Recursively Enumerable (RE) Language

  • Also known as Turing Recognizable or Partially Decidable.
  • A language L is RE if there exists a Turing Machine M such that if a string w ∈ L, the TM will eventually halt and accept it.
  • The Catch: If w ∉ L, the TM might halt and reject, OR it might loop indefinitely. It is not guaranteed to give a "No" answer.

Recursive (REC) Language

  • Also known as Turing Decidable or simply Decidable.
  • A language L is Recursive if there exists a Turing Machine M that will always halt.
  • If w ∈ L, it halts and accepts. If w ∉ L, it halts and rejects. There are no infinite loops.
Figure 1: Every Recursive language is also Recursively Enumerable, but not vice-versa.

2. Decidability and Undecidability

In computability theory, a "problem" is typically phrased as a question that requires a "Yes" or "No" answer for a given input.

Decidable Problems

  • A problem is decidable if there exists an algorithm (a Turing Machine) that always halts with the correct Yes/No answer for every possible input.
  • Example: "Is string w a palindrome?" – Decidable, because we can write a program that definitively checks this and stops.

Undecidable Problems

  • A problem is undecidable if no such algorithm can ever exist. It is mathematically impossible to write a program that will always halt and give the correct answer for *every* input.
  • Note: This doesn't mean we can't solve it for *some* inputs. It means we cannot guarantee a solution for *all* infinite possible inputs.
  • Example: "Will this piece of code run forever or eventually stop?" (This is the Halting Problem).

3. The Halting Problem

The Halting Problem, famously proven by Alan Turing in 1936, is the foundational example of an undecidable problem.

The Question: Given a description of a Turing Machine M and an input string w, is it possible to write a universal program H that can definitively say whether M will eventually halt on w, or loop forever?

Proof by Contradiction

We prove it is undecidable by assuming it IS decidable, and then showing this leads to a logical paradox.

  1. Assume there is a magic machine H. If we feed it (M, w), it prints "Halts" or "Loops".
  2. Now, let's build a new, tricky machine D. Machine D uses machine H internally as a subroutine.
  3. Machine D takes a program description M as input, and asks H: "What does M do when fed its own description M?"
  4. Machine D then does the exact opposite of what H predicts:
    • If H says M halts, D enters an infinite loop.
    • If H says M loops, D immediately halts.
  5. The Paradox: What happens if we feed machine D its own description? (We ask D(D)).
    • If D halts, H predicts "Halts", causing D to loop forever!
    • If D loops, H predicts "Loops", causing D to halt!
Figure 2: The construction of Machine D leading to the Halting Problem contradiction.

Because the existence of H leads to an impossible paradox, H cannot exist. Therefore, the Halting Problem is Undecidable.

4. Rice’s Theorem

Rice's theorem is a powerful generalization of the Halting Problem. It states a fundamental limit on what we can computationally verify about computer programs.

Rice's Theorem: Every non-trivial property of the languages recognized by Turing Machines is undecidable.

Trivial vs. Non-Trivial Properties

  • Trivial Property: A property that is either true for all RE languages, or false for all RE languages.
    Example: "Is the language recognized by a TM?" (Yes, by definition, this is trivially true for all TMs). Trivial properties are decidable.
  • Non-Trivial Property: A property that is true for some RE languages, but false for others.
    Examples: "Does the TM accept the empty string?", "Is the language finite?", "Does the TM accept more than 5 words?".

Implication: You cannot write an algorithm that takes the source code of an arbitrary program and perfectly analyzes what that program actually does (its semantic behavior) beyond executing it.

5. Post Correspondence Problem (PCP)

The PCP, introduced by Emil Post, is an undecidable decision problem that does not directly involve Turing Machines. It is often framed as a puzzle using "dominoes" or tiles.

The Problem Definition

You are given a finite set of tiles. Each tile has a string printed on the top and a string printed on the bottom. You can use an infinite supply of each tile type.

The Question: Is there a sequence of tiles (with repetitions allowed) such that if you read the top strings left-to-right, it perfectly equals the bottom strings read left-to-right?

Figure 3: PCP Example. Top string and Bottom string successfully match!

Significance of PCP

  • Despite appearing as a simple string-matching game, PCP is rigorously proven to be Undecidable. There is no algorithm that can determine if an arbitrary set of dominoes has a winning sequence.
  • Because it is easier to work with than Turing Machines, PCP is heavily used to prove the undecidability of other problems (via reduction), especially problems related to Context-Free Grammars (e.g., "Is this CFG ambiguous?").
  • Modified PCP (MPCP): A variation where the sequence must start with the very first tile (Tile 1). MPCP is also undecidable and is often used as an intermediate step to prove PCP is undecidable.
